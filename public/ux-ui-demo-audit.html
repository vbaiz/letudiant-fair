État des lieux — où on en est

La plateforme a une vraie identité de marque (rouge tomate, bleu piscine, typographie éditoriale, stripes multicolores). Le système est réalisé à 75% : tokens solides, composants Button/Card/Input cohérents, navigation propre. Les 25% restants — empty states, transitions, états de chargement, micro-interactions — sont précisément ce qui sépare une app fonctionnelle d'une app mémorable en démo.

🔴 Critical — à corriger avant la démo

Ces points seront visibles sur scène. Tout corriger prend ~2h.

1. Console.log pollués dans la page Discover 10 min

app/(student)/discover/page.tsx:494-601, 836

Des dizaines de console.log de debug, dont un dans le JSX ligne 836 ({'{console.log(...)}'}) qui s'exécute à chaque render. La DevTools ouverte pendant la démo trahit immédiatement le côté brouillon.

Fix. Grep console.log sur tout app/(student)/, supprimer ou remplacer par un logger conditionné à NODE_ENV.

2. Données hardcodées dans la page Fair 15 min

app/(student)/fair/page.tsx:51-54

"Salon de l'Orientation — Paris", "15 avril 2026", "Palais des Congrès" sont en dur dans le JSX. Si la démo se fait sur un autre événement, l'incohérence est immédiatement visible.

Fix. Charger depuis Supabase via eventId de l'URL, fallback skeleton sinon.

3. Variable CSS --le-red-dark non définie 5 min

app/globals.css:250 (utilisée), :root (manquante)

La variable est référencée pour les hover states mais n'existe pas dans :root. Le navigateur tombe en couleur par défaut → hovers cassés sur les boutons primaires.

Fix. Ajouter --le-red-dark:#B91C1C; --le-red-light:#FEE2E2; dans :root.

4. Auto-reset du formulaire booth après 5s 10 min

components/features/BoothCaptureFormV2.tsx:224-226

Après l'affichage des recommandations, setTimeout(resetForm, 5000) efface l'écran sans prévenir. En live, le présentateur lit les recos puis l'écran se vide pendant qu'il parle.

Fix. Supprimer le timeout, ajouter un bouton explicite "Nouveau scan".

5. Empty state exhibitor dashboard manquant 20 min

app/exhibitor/dashboard/page.tsx:214-221

Pour un compte fraîchement créé, les KPI affichent 0 sans message d'amorçage. Si la démo commence par un compte vierge, le dashboard a l'air cassé.

Fix. Si stats?.totalScans === 0, afficher empty state branded : "Aucun scan pour le moment — partagez votre QR pour commencer" + CTA imprimer QR.

6. Inline styles bloquant le responsive admin dashboard 45 min

app/admin/dashboard/page.tsx:71-104, 399

gridTemplateColumns: 'repeat(4, 1fr)' ne s'adapte pas en dessous de 1280px. Sur projecteur 1080p ou tablette, les cards KPI se chevauchent.

Fix. Remplacer par repeat(auto-fit, minmax(220px, 1fr)). Tester sur 1366×768 (résolution typique de salle de conférence).

7. Bouton "+ Ajouter" comparateur sans état désactivé 10 min

app/(student)/compare/page.tsx:70

Quand 3 écoles sont déjà sélectionnées (limite max), le bouton reste cliquable visuellement. Un clic ne fait rien → utilisateur perdu.

Fix. Ajouter disabled + opacity:0.4; cursor:not-allowed + tooltip "Maximum 3 écoles".

🟢 Wow factor — les 5 moments qui marquent

Si on a 4h de plus, c'est ici qu'il faut les mettre. Chaque moment est conçu pour être un screenshot mental que les jurés vont garder.

⚡ Moment 1 — La "control room" admin Jour J 1h

app/admin/dashboard/page.tsx:506-507 (ReferenceLine "Maintenant")

Le ReferenceLine indiquant l'heure courante existe déjà mais est en pointillés statiques. Transformer en ligne pulsante rouge avec @keyframes pulse + un badge "● LIVE" qui clignote.

Effet démo. "Vous voyez, on est en direct sur le salon. Là il est 14h32, les visites pic sont à 15h…" — instantanément lisible comme dashboard opérationnel.

🎬 Moment 2 — Card flip 3D sur la page Discover 15 min

app/globals.css:691-723 (animation déjà définie, sous-utilisée)

Le flip 3D cubic-bezier(0.68,-0.55,0.27,1.55) est déjà codé mais peu visible. Ajouter un hint pulsant 💡 les 3 premières secondes pour signaler "tappez pour voir le verso".

Effet démo. Le présentateur tape, la card pivote en 3D, voilà — un truc qui n'existe pas dans 90% des apps Tailwind génériques.

🎯 Moment 3 — QR code "vivant" exhibitor 20 min

app/exhibitor/dashboard/page.tsx:290-292

Ajouter une animation breathe sur le QR code :

@keyframes breathe {
  0%,100% { filter: drop-shadow(0 0 0 rgba(236,31,39,0)); }
  50%     { filter: drop-shadow(0 0 12px rgba(236,31,39,.4)); }
}
Effet démo. Le QR n'est plus une image figée — il "respire" en rouge L'Étudiant. Le ressenti passe de "image générée" à "interface active".

🚀 Moment 4 — Progress steps animés sur le formulaire booth 30 min

components/features/BoothCaptureFormV2.tsx:36 (5 steps existants)

Aujourd'hui le passage entre les 5 steps (orientation → grades → specialty → preferences → recommendations) n'a aucun indicateur visuel. Ajouter une stepper bar avec :

  • Steps complétés en rouge plein avec checkmark animé qui se dessine (SVG stroke-dasharray)
  • Step actuel en rouge avec ring pulsant
  • Steps à venir en gris
  • Ligne de progression qui se remplit avec cubic-bezier(0.34,1.56,0.64,1) (bounce)
Effet démo. Le moment où on remplit le formulaire devient visuel et satisfaisant — c'est le cœur du parcours booth, autant qu'il soit beau.

🎉 Moment 5 — Empty states illustrés (vs emojis) 1h

discover/page.tsx, schools/page.tsx, saved/page.tsx

Aujourd'hui les empty states sont des emojis 🎓 🔍. Créer 3 illustrations SVG branded : "Pas encore de favoris" (carnet ouvert vide), "Aucun résultat" (loupe avec ?), "Pas de RDV" (calendrier vide). Style line-art en rouge L'Étudiant.

Effet démo. C'est ce que les studios design (Linear, Stripe, Vercel) font systématiquement. Coût : ~1h pour 3 SVG. Bénéfice : positionne la plateforme dans la même catégorie visuelle.
Si tu ne peux en faire qu'un. Le Moment 1 (control room admin). Pourquoi : c'est la seule slide où l'on peut dire "voilà ce que les organisateurs voient en temps réel pendant le salon" — un angle B2B que les autres équipes Claude n'auront pas pensé à packager.

Demo flow — script 3 minutes

L'objectif : montrer les 3 rôles (étudiant, école, organisateur) en un parcours fluide qui raconte une histoire — pas une liste de features.

1
00:00 – 00:25 — L'accroche

Page d'accueil étudiant + swipe Discover

Ouvrir directement sur Discover. Faire le card flip 3D sur une école. Swiper droite → toast branded "Ajouté à mes favoris" qui slide-up.

À l'écran : Card flip animé, toast en bas avec barre rouge latérale, compteur favoris qui s'incrémente avec bounce.
2
00:25 – 00:55 — La promesse data

Page Saved → ouvrir un dossier → montrer alignment score

Cliquer sur un dossier favori. Montrer la barre de progression 5 étapes (Brouillon → En cours → Envoyé → Entretien → Décision) et le badge alignment 🎯 "Très aligné".

Phrase clé : "On capture l'intent de l'étudiant et on le matche avec les écoles. Score de 0.87 ici — Jaccard sur les filières."
3
00:55 – 01:30 — Le côté école

Switch vers exhibitor → QR code "respirant" + scan

Login exhibitor. Le QR respire en rouge. Démo : un étudiant scanne, le formulaire booth s'ouvre avec stepper animé, à la fin → recommandations personnalisées avec fit score.

Phrase clé : "30 secondes au stand, l'école repart avec un lead qualifié, l'étudiant repart avec 3 recos personnalisées."
4
01:30 – 02:15 — Le control room

Switch admin dashboard onglet "Jour J"

La ligne "Maintenant" pulse en rouge, badge ● LIVE. Visites en temps réel par heure, top écoles, heatmap fréquentation. C'est le moment qui doit faire dire "ah ça c'est différent".

Phrase clé : "Et pendant que tout ça se passe, l'organisateur voit le salon vivre en direct. Données rafraîchies toutes les 30 secondes."
5
02:15 – 03:00 — La couche data

Onglet Segments → ouvrir le HTML playbook data architecture

Montrer les 4 profils étudiants, ouvrir le data-architecture-playbook.html sur une page. Conclusion : "On ne se contente pas de digitaliser — on construit un actif data."

Phrase clé : "Avant le salon, pendant, après. 22 tables, 4 algorithmes de scoring, un actif data qui s'enrichit à chaque salon."
Pourquoi ce flow gagne. Il enchaîne 3 rôles distincts en 3 minutes — la plupart des démos Claude restent monorôle. La transition admin "Jour J" + le HTML playbook positionne le projet comme plateforme, pas appli.
